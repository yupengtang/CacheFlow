CacheFlow

Motivation

The problem with sequential decoding

Stock llama.cpp processes one request at a time. When serving multiple users, requests queue behind each other, GPU utilization stays low during the memory-bound decode phase, and the KV-cache allocation strategy wastes memory through fragmentation. I profiled the full stack over long generations (10K+ tokens) and 1–16 concurrent requests to find out exactly where time was being spent.

Two bottlenecks dominated: KV-cache memory reads during attention (the GPU stalls waiting on DRAM) and the inability to amortize fixed overhead across requests. Everything else — the matmuls, the sampling, the tokenization — was already fast.

profiling

Memory bandwidth bound

During autoregressive decode, each token reads the full KV-cache for that sequence. With 32 layers × 128-dim heads, a 4K-context request reads ~256 MB per step. The compute-to-memory ratio is terrible — the GPU spends most of its time waiting on memory.

profiling

Fragmented KV allocation

Contiguous per-sequence allocation means each request reserves its maximum possible length upfront. Short requests waste the tail, completed requests leave holes, and eventually the allocator can't serve new requests despite plenty of total free memory.

profiling

Sequential request handling

Without batching, the GPU processes one decode step for one request per iteration. The attention kernel's arithmetic intensity is low for a single sequence — batching multiple sequences multiplies the useful compute per memory transaction.

Architecture

System overview

CacheFlow sits between the request queue and the model forward pass. The scheduler decides which requests to batch, the block manager handles KV-cache allocation, and custom CUDA kernels execute paged attention over the non-contiguous block layout.

cuda

Paged Attention V1 / V2

V1 assigns one warp per query head — good up to ~8K context. V2 partitions across the sequence dimension with a log-sum-exp reduce, scaling to 32K+ contexts. Both handle the block-table indirection to read from non-contiguous physical blocks.

c++

Continuous batching

The scheduler adds and removes sequences at each decode step. Prefilling a new request and decoding existing ones happen in the same batch. No sequence waits for others to finish.

c++

Block-based KV-cache

Fixed-size blocks with a block table mapping. Eliminates external fragmentation. Copy-on-write for beam search. Prefix sharing via reference-counted blocks. A slab allocator pre-allocates the full pool at startup.

cuda

Cache defragmentation

A compaction kernel moves scattered blocks to contiguous positions, improving TLB locality and reducing the working set footprint for subsequent attention passes. Triggered when external fragmentation exceeds a threshold.

At 8 concurrent requests the throughput is 6.2× the single-request baseline — roughly 1.8× what stock llama.cpp achieves at the same concurrency. Beyond 12 concurrent the gains taper: memory bandwidth saturates and the scheduler starts swapping sequences to CPU.

The variance reduction is most visible in long-horizon generation. Over 10K output tokens with 8 concurrent requests, the standard deviation of per-step latency dropped from 52.7 ms to 10.8 ms (−34%) — the block allocator and defragmentation kernel keep memory access patterns stable as sequences grow.

Technical Approach

What I changed and why

scheduler/
scheduler.cpp

Concurrency-aware scheduler with preemption

The scheduler maintains three queues (waiting, running, swapped) and implements continuous batching: it adds new sequences and removes finished ones at each decode iteration. When GPU memory is insufficient, it preempts the lowest-priority sequence — either swapping its KV blocks to CPU or dropping them entirely for recomputation later. The scheduling policy is pluggable (FCFS, shortest-remaining, priority).

kv_cache/
block_manager.cpp

Block-based KV-cache with copy-on-write

Each physical block stores KV pairs for a fixed number of token positions. A block table per sequence maps logical positions to physical blocks, so sequences don't need contiguous memory. Reference counting enables copy-on-write for beam search — forked sequences share blocks until one of them writes. The fragmentation analyzer tracks both internal fragmentation (partially filled blocks) and external fragmentation (free-list scatter).

cuda/
paged_attention.cu

CUDA kernel-level memory optimization

The paged attention kernel gathers K/V from non-contiguous blocks via the block table. Within each block, memory access is coalesced along head_dim (threads in a warp read consecutive floats). Two variants: V1 for short context (one warp per head, shared-memory QK accumulation) and V2 for long context (partitioned across the sequence dimension with a log-sum-exp correction in the reduce step). The compaction kernel periodically defragments the physical layout, improving TLB hit rates for subsequent passes.

kv_cache/
prefix_cache.cpp

Prefix-aware KV reuse

A trie indexed by token IDs stores references to previously computed KV blocks. When a new request shares a prefix with a cached sequence (common with system prompts), the scheduler skips the shared prefix's compute and reuses the cached blocks directly. LRU eviction frees unreferenced prefix blocks under memory pressure. With a 50% shared prefix, prefill time drops by roughly half.

profiler/
profiler.cpp

System-level profiling and instrumentation

Every decode step records time-to-first-token, time-per-output-token, KV-cache utilization, and fragmentation metrics. Welford's online algorithm computes running mean/variance without storing all samples. A histogram tracks the full latency distribution for percentile reporting. The optional trace writer outputs Chrome Trace JSON for timeline visualization in chrome://tracing.

Implementation

Selected kernel excerpts

Paged Attention V1 — per-block QK dot product

The inner loop iterates over physical blocks via the block table. Within each block, the warp computes Q·K for every cached token position. Lane-parallel reduction across head_dim, then warp-level max for the softmax numerics. 

for (int block_idx = warp_idx; block_idx < num_blocks;
     block_idx += num_warps) {

    const int phys_block = block_tables[
        seq_idx * max_blocks_per_seq + block_idx];

    for (int tok_off = 0; tok_off < BLOCK_SIZE; ++tok_off) {
        const int tok_pos = block_idx * BLOCK_SIZE + tok_off;
        if (tok_pos >= ctx_len) break;

        const float* k = key_cache +
            ((phys_block * num_kv_heads + kv_head) * BLOCK_SIZE
             + tok_off) * HEAD_DIM;

        float dot = 0.0f;
        for (int d = lane_idx; d < HEAD_DIM; d += WARP_SIZE)
            dot += q[d] * k[d];
        dot = warp_reduce_sum(dot) * scale;

        if (lane_idx == 0)
            smem_qk[block_idx * BLOCK_SIZE + tok_off] = dot;
        qk_max = fmaxf(qk_max, dot);
    }
}

Copy-on-write — shared block deduplication

When a beam search fork writes to a shared block, COW checks the reference count. If the source block is unshared (ref_count == 1), it skips the copy entirely. Otherwise it allocates a new block, copies the data, and decrements the source's refcount. 

BlockId BlockManager::copy_on_write(
    BlockId src_block, SeqId new_seq, DeviceType device)
{
    std::lock_guard<std::mutex> lock(mu_);
    auto& src = blocks_for(device)[src_block];

    if (src.ref_count == 1) {
        // sole owner — no copy needed
        seq_block_map_[new_seq].emplace_back(device, src_block);
        return src_block;
    }

    BlockId new_bid = pop_free(free_list_for(device));
    if (new_bid == INVALID_BLOCK) return INVALID_BLOCK;

    auto& dst = blocks_for(device)[new_bid];
    dst.ref_count  = 1;
    dst.num_filled = src.num_filled;
    --src.ref_count;

    cow_pairs_.push_back({src_block, new_bid, device});
    seq_block_map_[new_seq].emplace_back(device, new_bid);
    return new_bid;
}

mkdir build && cd build

./benchmarks/bench_latency   -n 100 -p 128
